# Отчёт по нагрузочному тестированию и бенчмаркингу запросов

## Как проводилось тестирование

### Нагрузочное тестирование
Тестирование проводилось в трёх сценариях:
- **Insert-only** — только вставка записей (50, 75, 100, 125 VUs)
- **Update-only** — только обновление записей (100 VUs, ramp up)
- **Mixed (Insert+Update)** — смешанная нагрузка (ramp up до 300 VUs)

Объём данных в таблице на момент начала тестов: **300 млн**, **600 млн**, **1 млрд записей**.

### Бенчмаркинг запросов
- На каждой итерации (300 млн → 600 млн → 1 млрд) данные заполнялись **батчами по 1 млн записей**.
- Среднее время загрузки одного батча:
  - **300 млн:** 30 сек
  - **600 млн:** 120 сек
  - **1 млрд:** 300 сек
- **Индексы создавались только после полной загрузки данных** (кроме первичного ключа, который сохранялся на всех этапах).
- Перед переходом к следующему объёму **все индексы (кроме первичного ключа) удалялись**
- Для каждого объёма выполнялись **5 прогонов** каждого запроса, фиксировались min/max/avg/median.
- Для read-теста (`accounted_for_bs_profile_id`) — выборка **100 случайных значений**.

### Размер таблицы (на 1 млрд записей)
- **До создания индексов:** ~520 ГБ
- **После создания всех индексов:** ~1.3 ТБ

# Общий анализ и выводы

## Нагрузочное тестирование

- Insert‑тесты стабильны до 1 млрд записей: p(95) ≈ 55–76 мс, RPS растёт линейно.
- При 600 млн есть редкие выбросы, но общая успешность высокая.
- Update‑тесты до переработки деградировали:
  - при 600 млн RPS падал до 3.84, p(95) до 57 секунд.
  - причина — долгий запрос выборки перед обновлением.
- Смешанная нагрузка (Insert + Update):
  - до 300 млн — стабильно,
  - при 600 млн — резкое падение из‑за выборки.
- После переработки (1 млрд):
  - p(95) = 63 мс,
  - RPS = 674,
  - 100% успеха.

## Бенчмаркинг запросов

### Линейно масштабируемые запросы

- Фильтрация по индексированным полям: 0.7–3 мс на всех объёмах.
- Медиана практически не меняется.

### Проблемный запрос — GROUP BY bs_quota_id

- 300 млн: ~2.7 сек  
- 600 млн: ~114 сек  
- 1 млрд: ~208 сек  

### Общая пагинация

- Медиана 115–120 мс на всех объёмах.
- Редкие скачки до ~290 мс.

### Read‑тест

- Среднее 2.07–2.18 мс, медиана ~2 мс.
- Индекс полностью покрывает запрос.

## Влияние индексов на размер

- При 1 млрд записей таблица выросла с 520 ГБ до 1.3 ТБ.
- Это ожидаемо для широкой таблицы с составными индексами.

# Итоговые выводы

- Индексация выполнена качественно: точечные чтения быстрые даже на 1 млрд записей.
- Единственное узкое место — GROUP BY по `bsQuotaId`.
- Рост объёма данных не ухудшает индексированные запросы.
- Переработка update‑логики восстановила линейную масштабируемость.
